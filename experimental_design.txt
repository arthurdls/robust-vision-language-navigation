EXPERIMENTAL DESIGN: LTL-Guided Neuro-Symbolic UAV Control with
Goal Adherence Monitoring (PRELIMINARY RESULTS)
======================================================================

SCOPE NOTE
----------
This document describes a small-scale, single-task evaluation intended
to produce PRELIMINARY RESULTS for the paper. A larger, multi-task
evaluation suite is left to future work. The current run is one
difficult long-horizon task in one map, executed once per condition.
Statistical tests are not appropriate at this sample size; the
evidence is qualitative (per-condition outcome plus video review)
and is framed accordingly throughout.


1. SYSTEM OVERVIEW (FULL PROPOSED METHOD)
======================================================================

Pipeline: NL -> LTL-NL (via LLM + Spot automaton) -> Subgoal sequence
          -> SubgoalConverter -> OpenVLA control loop
          + Goal adherence monitor (local + global prompts, 3x3 image grid,
            text diary, displacement tracking, corrective-action history)
          + Convergence detection + Post-Convergence Supervisor Mode
            (corrective commands up to budget)
          + Stall detection (ask_help when plateaued)

Key components:
  - LTL Planner: LLM generates LTL-NL formula with pi predicates.
    Spot translates to deterministic monitor automaton. State machine
    advances through subgoals as each is confirmed complete.
  - SubgoalConverter: Rewrites planner predicates into short OpenVLA
    instructions. Strips visual detection conditions ("until you see X")
    and converts spatial proximity ("until near X") to approach commands.
  - Goal adherence monitor: Periodic VLM checkpoints during execution.
    Local prompt: 2-frame comparison ("what changed?").
    Global prompt: sampled 3x3 grid + text diary + displacement +
    completion history ("how complete?").
    Convergence prompt: evaluate on drone stop, issue corrective
    command if incomplete. Past correctives for this subgoal are
    surfaced to the convergence VLM in one of two equivalent forms,
    depending on whether the condition's convergence prompt displays
    the diary:
      * Diary-using conditions (C0, C2, C6): each issued corrective
        drops a `[CONVERGENCE @ step N]: corrective issued
        (<diagnosis>, <pct>% complete) -- "<instruction>"` marker into
        the running diary, so the convergence VLM sees past correctives
        interleaved with frame observations and can localise each
        corrective in the temporal record.
      * Non-diary conditions (C4, C5): the convergence prompt instead
        carries an explicit oldest-first corrective-action history
        block (same fields), since these ablations omit the diary by
        design. C5 additionally disables the local VLM, so its diary
        contains only the markers (still persisted to
        diary_summary.json for record-keeping but not shown to the VLM).
    Both forms ship the same directive: if completion has not improved
    since a recent correction, switch axes for this correction
    (e.g., altitude or a different turn direction) instead of
    reissuing the same command. C1 and C3 have no monitor and no
    convergence cycle and are therefore unaffected.
  - OpenVLA frame: the proprioceptive state sent to OpenVLA is
    rebased to [0, 0, 0, 0] at the start of every new low-level
    instruction (initial converted instruction or any monitor-issued
    correction). The diary, displacement vector, and corrective-action
    history remain subgoal-relative; only the OpenVLA proprio is
    instruction-relative.
  - OpenVLA Server: Vision2Seq model fine-tuned on first-person drone
    navigation. Takes image + proprioceptive state + instruction,
    outputs action poses.

Default parameters (from project configuration):
  - LLM model: gpt-4o (LTL planner; SubgoalConverter; C2's LLM text
    decomposer, which functionally replaces the LTL planner). Cheaper
    and adequate for structured text-to-text decomposition.
  - VLM / monitor model: gpt-5.4 (goal adherence monitoring: local,
    global, convergence; C6 text-only global also uses gpt-5.4).
  - SubgoalConverter uses the LLM model (gpt-4o), not the VLM, since
    subgoal conversion is a text-to-text task with no image input.
  - C2's LLM text decomposer uses the LLM model (gpt-4o) so that C0
    and C2 share the same model at the planning stage; the only
    difference is LTL + Spot vs. flat numbered list.
  - Max steps per subgoal: 500
  - Diary check interval: every 10 frames
  - Max corrections per subgoal: 20
  - Stall window: 10 checkpoints (hardware override: 5)
  - Stall threshold: 0.05 (completion spread)
  - Convergence: position delta < 3.0 cm AND yaw delta < 1.0 deg
    for 10 consecutive steps
  - Diary mode: frame-based for simulation eval (10-frame interval);
    run_hardware forces time-based (3 s interval)

Simulation: Unreal Engine via UnrealCV gym. Single map:
DowntownWest (urban). See Section 4a for details.


2. CONDITIONS (7 TOTAL: 1 FULL SYSTEM + 6 BASELINES)
======================================================================

CONDITION 0: FULL SYSTEM (Proposed Method)
------------------------------------------
Pipeline: NL -> LTL-NL -> Subgoal -> SubgoalConverter -> OpenVLA
          + Goal adherence monitor (local + global with 3x3 grid + text diary
            + displacement) + Convergence corrections + Stall detection
Why: This is the proposed method. All other conditions ablate or
     replace components of this pipeline.

2a. REPLICATION MAP (entry points only)
----------------------------------------
These names identify where each condition is run in the codebase. They
do not specify the scientific definition of the conditions, which is
given by the pipeline descriptions above.
  C0: run_integration.py
  C1: run_condition1_naive.py
  C2: run_condition2_llm_planner.py
  C3: run_condition3_open_loop.py
  C4: run_condition4_single_frame.py
  C5: run_condition5_grid_only.py
  C6: run_condition6_text_only.py

CONDITION 1: NAIVE END-TO-END VLA
---------------------------------
Pipeline: Full NL instruction -> OpenVLA
          (no LTL decomposition, no subgoal conversion, no monitoring)
Procedure: The entire long-horizon instruction is passed directly to
  OpenVLA as a single string. There is no LTL decomposition, no goal
  adherence monitoring, and no supervisor corrections. The low-level
  loop runs until convergence (drone stops moving).
Why: Demonstrates that the base OpenVLA model cannot handle a
  long-horizon, multi-step instruction on its own. Establishes that
  task decomposition is necessary for this class of instruction.
Expected outcome: Very low success. OpenVLA will likely attempt the
  first mentioned action and then stop or drift.

CONDITION 2: LLM SEQUENTIAL PLANNER (No LTL)
---------------------------------------------
Pipeline: NL -> LLM text-to-text decomposition -> Sequential subgoals
          -> SubgoalConverter -> OpenVLA
          + Goal adherence monitor (full, same as Condition 0)
Procedure: The same language model (gpt-4o, identical to C0's LTL
  planner) decomposes the instruction into a numbered list of
  subgoals (text only, no LTL formula, no automaton). Subgoals are executed in order with the
  same closed-loop monitor and corrections as Condition 0. The next
  subgoal starts when the monitor reports completion. There is no
  formal state machine.
Why: SECONDARY COMPARISON. Included as a representative of existing
  LLM-planner approaches (SayCan, Inner Monologue, Code-as-Policies)
  applied to UAV navigation. Isolates the LTL formalism by replacing
  the LTL planner + Spot automaton with a plain LLM text decomposition
  while keeping everything else identical.
  NOTE: The current task is purely sequential (Section 4b), so a
  flat numbered list is an adequate representation. The expressiveness
  advantage of LTL (OR branches, concurrent obligations, conditional
  dependencies) requires non-sequential task structures, which are
  left to future work. This comparison is therefore expected to yield
  a small or null difference on the current task.
Expected outcome: Similar performance to Condition 0 on a purely
  sequential task. A flat list and a linear-chain LTL formula are
  functionally equivalent for strict A-then-B-then-C orderings. Any
  observed difference would reflect decomposition quality rather than
  formal expressiveness.

CONDITION 3: OPEN-LOOP LTL (No Monitor, No Corrections)
--------------------------------------------------------
Pipeline: NL -> LTL-NL -> Subgoal -> SubgoalConverter -> OpenVLA
          (NO goal adherence monitoring, NO convergence corrections)
          Advance to next subgoal on convergence (drone stops = done).
Procedure: Full LTL-based planning and subgoal conversion as in
  Condition 0, but the control loop has no monitor. When the drone
  stops (convergence), the system treats the subgoal as done and
  advances. There is no completion check and no corrective commands.
Why: PRIMARY ABLATION AND HEADLINE COMPARISON. This is the single
  most important baseline in the experiment. C3 vs C0 directly tests
  the paper's core claim: that closed-loop goal adherence monitoring
  is necessary for reliable long-horizon execution. Both conditions
  share the same LTL planner, so any performance gap is attributable
  entirely to the monitoring and correction pipeline.
  Directly attacks the assumption in prior LTL + low-level-policy
  literature that low-level controllers execute perfectly. By showing
  how often OpenVLA stops prematurely (false convergence) and ruins
  the downstream LTL sequence, we justify the goal adherence
  monitoring and supervisor architecture.
Expected outcome: Planning is correct (LTL decomposition works) but
  execution fails. The drone stops short, treats the subgoal as done,
  and blindly advances, causing cascading failures on later subgoals.
SUCCESS METRIC NOTE: Convergence in C3 is self-fulfilling (the system
  always considers convergence as "success"), so the automated signal
  is not meaningful. Task success for C3 is determined by human video
  review, same as all other conditions (see Section 6d).

CONDITION 4: FRAME-BY-FRAME MONITOR (No Temporal Context)
----------------------------------------------------------
Pipeline: NL -> LTL-NL -> Subgoal -> SubgoalConverter -> OpenVLA
          + Single-frame VLM check (current frame + subgoal only)
          + Convergence corrections (using single-frame assessment)
Entry point: same integration harness as Condition 0, monitor set to
  the single-frame variant.
Procedure: At each checkpoint, the monitor sees only the current
  first-person frame and the subgoal text. There is no running text
  diary, no multi-frame grid, and no odometry block in the assessment.
  The protocol does not run a separate local two-frame narrative
  update, because that text is not used in this condition. Stall
  detection still uses completion history across checkpoints, but
  each per-checkpoint completion estimate is based on a single frame
  only. Convergence uses the same single-frame assessment. Monitor
  instructions describe only the inputs that are actually provided, so
  the model is not told to expect diary or displacement. Completion
  rules, orientation tolerance, and single-step correction guidance
  match the full system in strictness; only the available sensory
  context differs.
Why: Tests whether temporal context (the diary) is needed. A VLM
  looking at a single frame cannot distinguish between "stalled",
  "making slow progress", and "achieved", and has no way to track
  regression or calibrate completion percentage over time. Stall
  detection still fires based on the completion history, but each
  individual completion estimate lacks temporal grounding.
Expected outcome: Higher false positive rate (premature completion
  declarations) and degraded stall detection compared with the full
  system, particularly on approach/traversal subgoals where
  completion requires observing change over time.

CONDITION 5: IMAGE-GRID ONLY MONITOR (No Text Diary)
-----------------------------------------------------
Pipeline: NL -> LTL-NL -> Subgoal -> SubgoalConverter -> OpenVLA
          + Goal adherence monitor with 3x3 image grid but NO text diary
            and NO displacement data
          + Convergence corrections (using grid only)
Entry point: same integration harness as Condition 0, monitor set to
  the grid-only variant.
Procedure: Checkpoints use the same sampled 3x3 frame grid as the full
  system. Global and convergence assessments omit the running text diary
  and the displacement summary; the assessor sees only the subgoal and
  the grid. There is no local two-frame narrative stage in this
  condition, because those entries are not fed into global or
  convergence in this ablation. Monitor instructions describe only the
  inputs provided (subgoal + grid). Completion rules, orientation
  tolerance, and single-step correction guidance match the full system
  in strictness; only the available channels (visual sequence vs.
  visual + diary + odometry) differ.
Why: Tests whether VLMs can infer progress/completion from visual
  history alone without explicit textual state tracking. Isolates the
  contribution of the text diary and odometry data.
Expected outcome: VLM can detect gross completion (target clearly
  visible and close) but should struggle with precise distance
  estimation, subtle regression, and stall detection without the
  textual grounding from the diary. Performance expected between
  Condition 4 and Condition 0.

CONDITION 6: TEXT-ONLY GLOBAL ASSESSMENT (No Image Grid)
----------------------------------------------------------
Pipeline: NL -> LTL-NL -> Subgoal -> SubgoalConverter -> OpenVLA
          + Goal adherence monitor with text diary + displacement but NO
            3x3 image grid (global and convergence use a text-only LLM)
          + Convergence corrections (using text + displacement only)
Entry point: same integration harness as Condition 0, monitor set to
  the text-only global variant.
Procedure: Local checkpoints still build a short diary by comparing
  consecutive frames (this stage requires images). Global and
  convergence assessments use only the text diary, displacement, and
  prior completion summary via a text-only language model; no image
  grid is attached to those calls. Monitor instructions for those stages
  describe text-based reasoning from the diary and odometry. Completion
  rules, orientation tolerance, and single-step correction guidance are
  aligned with other conditions in strictness; obstacle-related guidance
  is omitted at the global and convergence stages because there is no
  direct image input there. Only the available sensory channel for those
  stages (text + odometry vs. text + odometry + grid) differs from the
  full system.

  NOTE: The paper should label this condition "Text-Only Global
  Assessment": a vision model still produces diary entries from frames,
  but completion and corrections at global and convergence are decided
  without an image grid.
Why: Tests whether the text diary and odometry data are sufficient
  for completion assessment, or whether visual confirmation from the
  image grid is needed. The text log might say "moved closer to the
  object" but only the camera can verify the drone arrived at the
  correct target (not a similar-looking one).
Expected outcome: Reasonable progress tracking but unreliable
  completion verification. The text diary tracks relative motion
  accurately but cannot visually confirm the drone is at the right
  target. Higher false positive rate on completion for tasks with
  similar-looking landmarks.


4. TASK DESIGN
======================================================================

4a. ENVIRONMENT
-----------------
Single simulation map (Unreal Engine, UnrealCV):

  DowntownWest: Urban setting with diverse landmarks.
    Buildings, vehicles, streetlights, traffic lights, pergolas,
    sculptures, people, dogs. Mixed commercial/residential.

The preliminary evaluation runs in this single map.

4b. TASK (1 task, 1 episode per condition)
--------------------------------------------
A single difficult, long-horizon, sequential navigation task is
used for this preliminary evaluation. The task exercises sequential
ordering, multi-landmark targeting, heading changes, and a final
landing maneuver, which together provide many opportunities for the
low-level controller to converge prematurely on incomplete subgoals.

Category: SEQUENTIAL (1 task, in DowntownWest)
  Pure sequential navigation. Strict A-then-B-then-...-then-F
  ordering with multi-step traversals.
  LTL (illustrative form for N=6 sequential predicates):
    F pi_6
    & (!pi_6 U pi_5) & (!pi_5 U pi_4) & (!pi_4 U pi_3)
    & (!pi_3 U pi_2) & (!pi_2 U pi_1)
  The exact formula generated at runtime is whatever the LTL planner
  emits for this instruction; the shape above is the canonical
  linear-chain encoding the planner targets.

Task: streetlamp_to_cars (file: first_task.json)
  "Go past the first streetlamp, then turn left until you see the
  black car, then go to the black car, then turn toward the red car
  and go to the red car. Finally, land in front of the traffic light
  pole on the right."
  Start: [-7500, 848, 396, -177]
  Source: tasks/rvln_tests/first_task.json
  Subgoal count: 6
    1. Go past the first streetlamp
    2. Turn left until the black car is in view
    3. Go to the black car
    4. Turn toward the red car
    5. Go to the red car
    6. Land in front of the traffic light pole on the right

This single task is run once per condition. There are no starting
position variants in the preliminary run.

4c. TASK JSON FORMAT
---------------------
The on-disk task JSON contains:
{
  "task_id": "streetlamp_to_cars",
  "instruction": "Go past the first streetlamp, then turn left ...",
  "initial_pos": [-7500, 848, 396, -177],
  "expected_subgoal_count": 6
}

`task_id` is used by the orchestrator to dedupe completed runs.
`expected_subgoal_count` is used to size the C1 (naive) total step
budget as `max_steps_per_subgoal * expected_subgoal_count` so C1 has
a fair chance to attempt every instruction step before its budget
runs out.

Per-run parameters not stored in the JSON (max_steps_per_subgoal,
max_corrections, diary_check_interval) are taken from the project
configuration described in Section 1.


5. EVALUATION METRICS
======================================================================

PRIMARY METRICS (reported in main results table):

M1. Task Success
  - Definition: Whether the final long-horizon goal was fully
    completed (all subgoals achieved in correct order).
  - Measurement: Binary per episode. Determined by human video review
    for ALL conditions (C0-C6). The experimenter watches the recorded
    episode and judges whether the instruction was fulfilled.
  - Report: Per-condition outcome (success / failure) on the single
    task. With one episode per condition there is no rate to report;
    confidence intervals are not meaningful and are not used.
  - Annotation: Recorded in results/annotations.csv (task_success column).

M3. Subgoal Completion Count
  - Definition: Number of individual subgoals successfully completed
    out of the number attempted.
  - Measurement: The experimenter watches the episode and records
    completed and total subgoals. Applies to all conditions (C0-C6).
    For C1 (naive, no explicit subgoals), the experimenter counts how
    many instruction steps were visibly accomplished.
  - Report: count completed / count attempted per condition.
  - Annotation: Recorded in results/annotations.csv (subgoals_completed
    and subgoals_total columns).

DIAGNOSTIC METRICS (reported in analysis section):

M4. Supervisor Correction Outcomes (Rescue Cases)
  - Definition: Of premature convergence events (drone stopped before
    subgoal completion), how many were successfully rescued by the
    supervisor's corrective commands?
  - Applicable to: Conditions 0, 2, 4, 5, 6 (not 1 or 3).
  - Measurement: Counts (rescued vs. total premature convergences)
    rather than rates, given the small sample.

M5. Convergence Assessment Errors
  - False Positive: Monitor declares subgoal complete but it is not
    (per the human success judgment from M1/M3).
  - False Negative: Monitor issues correction when subgoal was
    actually complete (causes unnecessary additional steps).
  - Applicable to: Conditions 0, 4, 5, 6.
  - Measurement: Compare monitor's automated decisions (from logs)
    against human ground-truth judgments (from annotations.csv).
    Reported as raw counts per condition for the single episode.

M6. Control Frequency / Latency
  - OpenVLA loop Hz: Frames per second of the control loop
    (excluding VLM checkpoint time in sync mode).
  - Monitor checkpoint latency: Wall-clock time per VLM checkpoint
    (local + global prompt pair).
  - Report: Mean and p95 within the single episode per condition.
  - Purpose: Demonstrates that VLM monitoring does not block drone
    control in async mode, and quantifies overhead in sync mode.

M7. Token / Inference Overhead
  - Total VLM/LLM calls per episode (broken down by type: local,
    global, convergence, planning, subgoal conversion).
  - Total input tokens per episode.
  - Total output tokens per episode.
  - Estimated API cost per episode.
  - Report: Per-condition values for the single episode.
  - Purpose: Demonstrates computational feasibility. Addresses
    reviewer concern about running VLM monitoring continuously.

M8. Episode Length
  - Total steps (frames) per episode.
  - Total wall-clock time per episode.
  - Report: Per-condition values for the single episode.
  - Purpose: Reveals whether monitoring adds overhead that slows task
    completion, even when it succeeds.


6. EXPERIMENTAL PROCEDURE
======================================================================

6a. PER-EPISODE PROCEDURE
---------------------------
1. Load task JSON (instruction, initial_pos).
2. Teleport drone to starting position. Wait for stabilization.
3. Reset OpenVLA model state.
4. Execute the pipeline for the active condition (see Section 2a for
   entry-point names).
5. Log all frames, diary entries, monitor decisions, corrections,
   VLM call records, and trajectory poses.
6. Record final outcome: success/failure, stop_reason, subgoal
   completion status.

6b. EPISODE COUNT
------------------
1 task x 1 run = 1 episode per condition
7 conditions x 1 = 7 total episodes

6c. RUN ORDER
---------------
- All 7 conditions are run on the same task and starting pose.
- The simulator is fully reset between episodes (teleport +
  stabilization wait), so there is no drift or carryover.
- Suggested order: run the primary baselines (C1, C3) first, then
  the full system (C0), then the secondary ablations (C2, C4, C5, C6)
  in any order.

6d. HUMAN VIDEO REVIEW (ALL CONDITIONS)
-----------------------------------------
ALL conditions (C0-C6) use human video review as the ground-truth
success criterion for M1 and M3. With only 7 episodes total, manual
review is trivial and eliminates the monitor-as-judge circularity
problem (where the monitor grades its own performance).

Procedure: After each episode completes, the experimenter watches the
recorded video and fills in results/annotations.csv with:
  - task_success: 1 if the full instruction was fulfilled, 0 otherwise
  - subgoals_completed: count of subgoals visibly completed
  - subgoals_total: count of subgoals the system attempted
  - notes: optional free-text observations

For M5 (monitor false positive/negative analysis), the human ground
truth from annotations.csv is compared against the monitor's automated
decisions from the episode logs. This is done post-hoc.

Analysis: Run scripts/analyze_annotations.py to compute the per-
condition outcome table from the completed CSV. With n=1 per
condition, the script outputs raw counts and outcomes only; no
aggregate rates or hypothesis tests are computed.


7. COMPUTE BUDGET
======================================================================

7a. TOKEN COST ESTIMATE
-------------------------
Empirical observation: total cost for the 7-episode preliminary run
is well under $5 (gpt-4o for LTL/SubgoalConverter, gpt-5.4 for
monitor). This is consistent with the per-condition breakdown below
and the small total episode count.

Per VLM checkpoint (local + global, gpt-5.4 monitor):
  - Local prompt: 1 system message + 2 images (224x224 each)
    Estimated: ~600 tokens (text) + ~1,700 tokens (2 images) = ~2,300
  - Global prompt: 1 system message + 3x3 grid image + diary text
    Estimated: ~800 tokens (text + diary) + ~4,000 tokens (grid image)
    = ~4,800
  - Total per checkpoint: ~7,100 input tokens

Per convergence assessment:
  - Similar to global: ~5,000 input tokens

Per episode (rough estimate, diary interval = 10 frames):
  - ~30 checkpoints per subgoal x 6 subgoals = 180 checkpoints
  - 180 x 7,100 = ~1,278,000 input tokens
  - Plus ~12 convergence assessments x 5,000 = ~60,000
  - Plus LTL planning + subgoal conversion (gpt-4o): ~5,000
  - Total per full-monitor episode: ~1,343,000 input tokens

IMPORTANT: These estimates should still be validated empirically on
the first C0 pilot run. Image token counts vary by provider and
resolution. With gpt-4o handling planning/conversion (cheaper) and
gpt-5.4 handling the monitor calls (heavier per-call), the bulk of
spend is on monitor VLM tokens.

Per-condition rough cost:
  - Full-monitor condition (C0, C2, C4, C5, C6): ~$0.50-$1.00 per
    episode at observed rates
  - Reduced-channel ablations (C4, C5, C6) cost slightly less due to
    smaller payloads
  - C1 (naive) and C3 (open-loop) use no monitor, costing only one
    LTL planning call (C3 only) plus subgoal conversion (C3 only);
    C1 has no LLM/VLM calls at all beyond OpenVLA inference

Observed total for 7 conditions x 1 episode each: under $5, which
allows generous headroom for re-runs and pilots.

7b. GPU COMPUTE
----------------
- OpenVLA inference: 1x GPU (CUDA, bfloat16)
- Simulation: Unreal Engine rendering (separate from GPU inference
  or on same machine with sufficient VRAM)
- VLM API calls: External API (no local GPU needed)

7c. WALL-CLOCK TIME ESTIMATE
------------------------------
Per episode: ~5-15 minutes (depends on number of corrections).
Total: 7 episodes x ~10 min avg = ~1-2 hours of run time.
With setup/teardown overhead and re-runs: ~3-4 hours.


8. ANALYSIS PLAN AND FRAMING
======================================================================

8a. ANALYSIS APPROACH (n=1 per condition)
-------------------------------------------
With one episode per condition (7 total), this is a single-case
demonstration, not a statistical study. Hypothesis tests, confidence
intervals, and significance levels are not used.

PRIMARY EVIDENCE: Per-condition outcome table.
  A 1-task x 7-condition row showing pass/fail and the subgoal
  completion count per condition. This is the main results table.

SUPPLEMENTARY EVIDENCE: Qualitative video analysis.
  - Annotated playback videos for each condition.
  - Trajectory plots and subgoal timelines.
  - For monitor-based conditions, monitor decision transcripts
    aligned with the human ground truth (M5).

PRIMARY COMPARISONS (descriptive only):
  - C0 vs C3: Monitoring ablation (headline comparison).
  - C0 vs C1: Decomposition necessity.

SECONDARY COMPARISONS (descriptive only):
  - C0 vs C2: LTL vs. flat LLM decomposition.
  - C0 vs C4: Single-frame monitor.
  - C0 vs C5: Grid-only monitor.
  - C0 vs C6: Text-only global assessment.

All comparisons are reported as outcome differences with qualitative
discussion. With n=1 per condition there is no statistical claim, and
the paper should not present p-values, confidence intervals, or
"significant difference" language.

All summaries computed by scripts/analyze_annotations.py.

8b. FRAMING (PRELIMINARY RESULTS)
-----------------------------------
The paper presents these runs as PRELIMINARY RESULTS supporting a
proof of concept. The framing should be explicit:
  - One difficult long-horizon task in one map, one episode per
    condition.
  - Outcomes are descriptive, not statistical.
  - The point of this evaluation is to demonstrate that the proposed
    pipeline can complete a task that ablations cannot, not to
    estimate a population success rate.
  - A larger, multi-task, multi-variant suite is identified as future
    work and is the natural follow-up extension.

If the full system (C0) succeeds where C3 (no monitoring) and C1
(no decomposition) fail, that pattern is informative even from a
single run, because failure modes for those baselines are expected
to be repeatable on this class of task.

        diary_summary.json carries the markers regardless of
        condition (useful for offline analysis).
      * "blk" (C4, C5): a separate oldest-first corrective-action
        block in the convergence prompt, since those conditions omit
        the diary by design. C5 additionally disables the local VLM
        (skip_local), so its persisted diary contains only the
        markers and no observation entries.
    C1 and C3 have no monitor and no convergence cycle, so neither
    form applies.

ABORT SEMANTICS:
  Monitor-based conditions (C0, C2, C4, C5, C6) use a uniform abort
  policy in batch mode. The runner classifies every subgoal termination
  into one of these stop_reason buckets, recorded in run_info.json so
  the M1/M3 analysis can keep them apart:

    - "monitor_complete" : monitor (or convergence VLM) said done.
                           SUCCESS.
    - "convergence"      : drone stopped moving, no monitor present
                           (C3 only). SUCCESS for C3, by design.
    - "ask_help"         : legitimate "would have asked the operator
                           in interactive mode". STRICTLY one of:
                             * stall detected for stall_window
                               checkpoints (monitor)
                             * max_steps per subgoal (frame mode)
                             * max_seconds per subgoal (time mode;
                               the time analogue of max_steps)
                             * max_corrections exhausted (monitor)
    - "monitor_error"    : monitor infrastructure failure (VLM
                           exception, JSON parse failure after
                           retry, no usable corrective after retry).
                           Distinct from ask_help so it can be
                           filtered out of headline metrics.
    - "no_image" / "no_response" / "empty_action" / "action_error" :
                           sim-side or OpenVLA-side infrastructure
                           failures.

  Frame mode (sim eval default) and time mode (hardware default) are
  strictly disjoint: the runner uses max_steps + check_interval (frame)
  XOR max_seconds + check_interval_s (time), never both. Mixing them is
  rejected by SubgoalConfig at construction time.

  C1 (naive) runs OpenVLA on the full instruction with no monitor and
  terminates on convergence (drone stops moving) or its total step
  budget; it never aborts via operator help. aborted is always false
  in run_info.json.

  C3 (open-loop LTL) deliberately has NO abort branch. Each subgoal
  terminates on convergence OR max-steps, and either outcome is
  treated as "subgoal done; advance the LTL automaton." This is by
  design: the entire point of C3 is to demonstrate that without a
  monitor, premature convergence (and step exhaustion) cascade
  through subsequent subgoals. Adding an abort would mask that
  cascade. The episode ends only when the LTL automaton reaches a
  terminal state or all planned subgoals have been processed.
  aborted is always false in run_info.json.

  All conditions write `aborted: <bool>` and a `stop_reason` to
  run_info.json. C1 and C3 always write `aborted: false`; the other
  five write `aborted: true` for any non-success stop_reason in
  the bucket list above.

  HARDWARE NOTE: mininav/interface.py uses its own ask-help logic and
  routes EVERY error class to operator help (an operator is present
  to triage). The strict ask_help vs. monitor_error split above is
  sim-only.

C4*: Single-frame global and convergence VLM, no diary, no local
     narrative stage. Convergence corrections and stall detection
     both operate on single-frame completion estimates (the stall
     logic still uses history across checkpoints). "SF" = instructions
     reference only the current frame and subgoal.
C5:  Image grid for global and convergence, no text diary, no
     displacement, no local two-frame narrative (not used by this
     ablation). "GO" = instructions reference the subgoal and the
     frame grid only.
C6:  Local two-frame comparison still produces diary text. Global and
     convergence use a text-only assessor (no image grid). In the
     paper, use the label "Text-Only Global Assessment" so readers do
     not infer that no vision is used anywhere. "TO" = text-only
     instruction set for global and convergence; obstacle guidance is
     omitted there because those stages have no images.

Instruction control: For monitor-based conditions, written
     instructions are matched in scope and strictness (what counts as
     done, orientation tolerance, one-step corrections) while varying
     only the information channels under study. Obstacle-related
     guidance is included when the assessor receives images (C0, C4,
     C5) and withheld for C6 at global and convergence, where there is
     no direct visual input.

Checkpoint scheduling (C6): Whether checkpoints are triggered by frame
     count or by elapsed time, global and convergence for Condition 6
     always follow the text-and-odometry protocol with no image grid,
     so the ablation is well defined across scheduling modes.


9. KEY ASSUMPTIONS AND LIMITATIONS
======================================================================

A1. Low-level UAV safety is handled separately.
    Physical collision avoidance, altitude bounds, and geofencing are
    assumed to be implemented via control barrier functions (CBF) or
    depth-map heuristics. This is standard in UAV systems and is
    orthogonal to the task-level planning contribution of this work.
    State this as a scope limitation in the paper.

A2. Simulation-only, single-task evaluation.
    All experiments run in a single Unreal Engine map (DowntownWest)
    on a single long-horizon task with a single starting pose. This
    is intended as a preliminary demonstration. Acknowledge the
    simulation-only and single-task scope in the paper and frame
    multi-environment, multi-task, and real-world validation as
    future work.

A3. OpenVLA as the fixed low-level policy.
    The system is designed to be policy-agnostic. OpenVLA is one
    instantiation. The contribution is the LTL + monitoring framework,
    not the specific VLA model.

A4. VLM accuracy for completion monitoring.
    The goal adherence monitor relies on VLM visual understanding to
    assess subgoal completion. VLM errors will propagate as incorrect
    completion decisions. Report observed errors qualitatively in M5
    for this episode.

A5. Simulation fidelity.
    UnrealCV provides photorealistic rendering but lacks dynamic
    obstacles, weather variation, and wind disturbances present in
    real-world flight.

A6. n=1 per condition.
    A single run cannot characterize variance. Outcomes are reported
    as a single observation per condition. The paper should not make
    population-level claims and should explicitly identify a larger
    evaluation suite as the planned follow-up.

11. PAPER FRAMING NOTES
======================================================================

- PRELIMINARY RESULTS framing: The paper should explicitly describe
  this evaluation as preliminary (single task, single starting pose,
  one run per condition) and identify a larger multi-task, multi-pose
  evaluation as planned future work.

- HEADLINE RESULT: Condition 3 vs. Condition 0. This is the primary
  comparison and should anchor the results section. Both share the
  same LTL planner, so any performance gap is attributable to the
  monitoring and correction pipeline. Even from a single run, this
  pattern is informative if C3 fails on a subgoal where C0's monitor
  successfully issues corrections.

- Condition 2 is a SECONDARY comparison. Frame it as a SOTA-
  representative baseline (cite SayCan, Inner Monologue, Code-as-
  Policies) but discuss the result honestly: C2 is expected to perform
  similarly to C0 on the current sequential task, because a flat
  numbered list is functionally equivalent to a linear-chain LTL
  formula for strict A-then-B-then-C orderings. State that evaluating
  LTL's expressiveness advantage requires non-sequential task
  structures, which is future work.

- Conditions 4, 5, 6 are design ablations that justify architectural
  choices in the monitoring pipeline. They can go in an ablation
  subsection or appendix if space is tight.

- Report M7 (token overhead) prominently to address feasibility
  concerns. Show that monitoring adds <X% wall-clock overhead in
  async mode, even though the comparison is per-condition rather
  than statistical.

- For C6, use the label "Text-Only Global Assessment" in all paper
  text and tables. Explicitly note in the methods section that C6's
  local diary entries are still generated by a VLM (2-frame
  comparison requires image input); only the global/convergence
  assessments are text-only.

- DO NOT report p-values, confidence intervals, or "statistically
  significant" language for this evaluation. With n=1 per condition,
  these are not meaningful and would mislead readers.
