EXPERIMENTAL DESIGN: LTL-Guided Neuro-Symbolic UAV Control with
Goal Adherence Monitoring
======================================================================

1. SYSTEM OVERVIEW (FULL PROPOSED METHOD)
======================================================================

Pipeline: NL --> LTL-NL (via LLM + Spot automaton) --> Subgoal sequence
          --> SubgoalConverter --> OpenVLA control loop
          + GoalAdherenceMonitor (local + global prompts, 3x3 image grid,
            text diary, displacement tracking)
          + Convergence detection + Post-Convergence Supervisor Mode
            (corrective commands up to budget)
          + Stall detection (ask_help when plateaued)

Key components:
  - LTL Planner: LLM generates LTL-NL formula with pi predicates.
    Spot translates to deterministic monitor automaton. State machine
    advances through subgoals as each is confirmed complete.
  - SubgoalConverter: Rewrites planner predicates into short OpenVLA
    instructions. Strips visual detection conditions ("until you see X"),
    converts spatial proximity ("until near X") to approach commands,
    flags out-of-distribution instructions.
  - GoalAdherenceMonitor: Periodic VLM checkpoints during execution.
    Local prompt: 2-frame comparison ("what changed?").
    Global prompt: sampled 3x3 grid + text diary + displacement +
    completion history ("how complete?").
    Convergence prompt: evaluate on drone stop, issue corrective
    command if incomplete.
  - OpenVLA Server: Vision2Seq model fine-tuned on first-person drone
    navigation. Takes image + proprioceptive state + instruction,
    outputs action poses.

Default parameters (from config.py):
  - LLM model: gpt-4o (planning, subgoal conversion)
  - VLM model: gpt-5.4 (goal adherence monitoring)
  - Max steps per subgoal: 300
  - Diary check interval: every 20 frames
  - Max corrections per subgoal: 15
  - Stall window: 3 checkpoints
  - Stall threshold: 0.05 (completion spread)
  - Convergence: position delta < 3.0 cm AND yaw delta < 1.0 deg
    for 10 consecutive steps

Simulation: Unreal Engine (DowntownWest environment), UnrealCV gym.
Single environment with diverse landmarks (buildings, trees, cars,
streetlights, pergolas, sculptures, people, dogs, traffic lights).


2. CONDITIONS (7 TOTAL: 1 FULL SYSTEM + 6 BASELINES)
======================================================================

CONDITION 0: FULL SYSTEM (Proposed Method)
------------------------------------------
Pipeline: NL --> LTL-NL --> Subgoal --> SubgoalConverter --> OpenVLA
          + GoalAdherenceMonitor (local + global with 3x3 grid + text diary
            + displacement) + Convergence corrections + Stall detection
Script: run_integration.py
Why: This is the proposed method. All other conditions ablate or
     replace components of this pipeline.

CONDITION 1: NAIVE END-TO-END VLA
---------------------------------
Pipeline: Full NL instruction --> OpenVLA
          (no LTL decomposition, no subgoal conversion, no monitoring)
Script: Modified run_ltl.py (skip planner, feed raw instruction)
Implementation: Pass the entire long-horizon instruction directly to
  OpenVLA as a single instruction string. No LTL decomposition, no
  goal adherence monitoring, no convergence corrections. Run the basic control
  loop with convergence detection only (stop when drone stops moving).
Why: Proves the base OpenVLA model cannot handle long-horizon,
  multi-step instructions. Establishes that task decomposition is
  strictly necessary.
Expected outcome: Very low success rate. OpenVLA will attempt the
  first mentioned action and then stop or drift.

CONDITION 2: LLM SEQUENTIAL PLANNER (No LTL)
---------------------------------------------
Pipeline: NL --> LLM text-to-text decomposition --> Sequential subgoals
          --> SubgoalConverter --> OpenVLA
          + GoalAdherenceMonitor (full, same as Condition 0)
Script: Modified run_integration.py (replace LTL planner with simple
  LLM decomposition, no Spot automaton, no formal state machine)
Implementation: Use the same LLM (gpt-4o) to break the instruction
  into a numbered list of subgoals (text-to-text, no LTL formula).
  Execute subgoals sequentially with the same goal adherence monitoring as the
  full system. Advance to the next subgoal when the monitor confirms
  completion. No automaton state tracking, no formal constraint
  enforcement.
Why: Isolates the contribution of LTL formalism. Shows that without
  the automaton state machine and formal constraint encoding:
  (a) The LLM may hallucinate impossible orderings or skip steps.
  (b) Temporal constraints ("never fly over X", "always stay above
      10m") have no formal enforcement mechanism.
  (c) Complex temporal logic (OR branches, concurrent obligations)
      cannot be expressed as a flat sequential list.
Expected outcome: Performs reasonably on simple sequential tasks but
  fails on tasks with temporal constraints, branching, or complex
  dependencies. The gap vs. Condition 0 should be largest on tasks
  with constraints.

CONDITION 3: OPEN-LOOP LTL (No Monitor, No Corrections)
--------------------------------------------------------
Pipeline: NL --> LTL-NL --> Subgoal --> SubgoalConverter --> OpenVLA
          (NO goal adherence monitoring, NO convergence corrections)
          Advance to next subgoal on convergence (drone stops = done).
Script: run_ltl.py (existing script, already implements this)
Implementation: Use the full LTL planner and SubgoalConverter, but
  run the basic OpenVLA control loop without any VLM monitoring. When
  the drone stops moving (convergence detected), assume the subgoal is
  achieved and advance the automaton to the next state. No completion
  assessment, no corrective commands.
Why: THE MOST IMPORTANT BASELINE. Directly attacks the assumption in
  prior LTL + low-level-policy literature that low-level controllers
  execute perfectly. By showing how often OpenVLA stops prematurely
  (false convergence) and ruins the downstream LTL sequence, we
  justify the entire goal adherence monitoring and supervisor architecture.
Expected outcome: Moderate planning accuracy (LTL decomposition works)
  but frequent execution failures. The drone will stop short, think
  it is done, and blindly advance, causing cascading failures in
  subsequent subgoals.

CONDITION 4: FRAME-BY-FRAME MONITOR (No Temporal Context)
----------------------------------------------------------
Pipeline: NL --> LTL-NL --> Subgoal --> SubgoalConverter --> OpenVLA
          + Single-frame VLM check (current frame + subgoal only)
          + Convergence corrections (using single-frame assessment)
Script: Modified run_integration.py (replace GoalAdherenceMonitor with
  a single-frame VLM query at each checkpoint)
Implementation: At each checkpoint interval, send ONLY the current
  camera frame and the subgoal text to the VLM. Ask: "Is the drone
  currently at the goal?" No diary history, no image grid, no
  displacement data, no completion percentage tracking. On convergence,
  use the same single-frame query to decide complete vs. stopped_short
  and issue corrections.
Why: Proves that temporal context (the diary) is necessary. A VLM
  looking at a single frame cannot distinguish between:
  - "Stalled" (no progress for many steps)
  - "Making slow progress" (gradually approaching)
  - "Achieved" (arrived at goal)
  Without history, the VLM has no way to detect stalls, track
  regression (moving away from target), or calibrate completion
  percentage over time.
Expected outcome: Higher false positive rate (premature completion
  declarations) and inability to detect stalls. Success rate lower
  than full system, particularly on approach/traversal tasks where
  completion requires observing change over time.

CONDITION 5: IMAGE-GRID ONLY MONITOR (No Text Diary)
-----------------------------------------------------
Pipeline: NL --> LTL-NL --> Subgoal --> SubgoalConverter --> OpenVLA
          + GoalAdherenceMonitor with 3x3 image grid but NO text diary
            and NO displacement data
          + Convergence corrections (using grid only)
Script: Modified run_integration.py (remove diary text and displacement
  from global/convergence prompts, keep image grid)
Implementation: Same checkpoint interval. Build the 3x3 grid of
  sampled frames as normal. But remove the running text diary entries
  and displacement [x, y, z, yaw] data from the global and convergence
  prompts. The VLM sees only the subgoal text and the image grid.
Why: Tests whether VLMs can infer progress/completion from visual
  history alone without explicit textual state tracking. Isolates the
  contribution of the text diary and odometry data.
Expected outcome: VLM can detect gross completion (target clearly
  visible and close) but struggles with:
  - Precise distance estimation (no displacement data)
  - Detecting subtle regression (text diary tracks "moved closer then
    moved away")
  - Stall detection (no completion percentage history to detect plateau)
  Performance between Condition 4 and Condition 0.

CONDITION 6: TEXT-ONLY DIARY (No Image Grid, "Blind LLM")
----------------------------------------------------------
Pipeline: NL --> LTL-NL --> Subgoal --> SubgoalConverter --> OpenVLA
          + GoalAdherenceMonitor with text diary + displacement but NO
            3x3 image grid (use standard LLM, not VLM)
          + Convergence corrections (using text + displacement only)
Script: Modified run_integration.py (remove image grid from global/
  convergence prompts, use LLM instead of VLM for global checks)
Implementation: Keep the local prompt (2-frame VLM query for diary
  entries, since it needs images). For global and convergence checks,
  send only the text diary and displacement data to a text-only LLM.
  No image grid in the global/convergence prompts.

  NOTE: The local prompt still requires a VLM call (it compares two
  frames to generate the diary entry). The key change is that the
  global assessment (completion %, complete/incomplete, corrections)
  is made by a text-only LLM reading the diary entries and
  displacement, with no image grid.
Why: Tests whether the text diary and odometry data are sufficient
  for completion assessment, or whether visual confirmation from the
  image grid is strictly required. The text log might say "moved
  closer to the object" but only the camera can verify the drone
  arrived at the correct target (not a similar-looking one).
Expected outcome: Reasonable progress tracking but unreliable
  completion verification. The text diary tracks relative motion
  accurately but cannot visually confirm the drone is at the right
  target. Higher false positive rate on completion for tasks with
  similar-looking landmarks.


3. TEMPORAL CONSTRAINT SUPPORT
======================================================================

The LTL planner supports temporal constraints (both avoidance and
maintenance) in addition to goal sequencing. The system handles all
constraint types that the Spot automaton can represent, using BDD
queries rather than formula parsing. Four constraint forms are
supported:

  Negative (avoidance) constraints:
  - G(!pi_X): Global avoidance. pi_X must never become true.
    Example: "never fly over building C" with G(!pi_3).
  - !pi_X U pi_Y: Scoped avoidance. pi_X must not become true until
    pi_Y is achieved. Example: "stay away from the red car until you
    reach the streetlight" with !pi_3 U pi_2.

  Positive (maintenance) constraints:
  - G(pi_X): Global maintenance. pi_X must always remain true.
    Example: "always stay above 10 meters" with G(pi_2).
  - pi_X U pi_Y: Scoped maintenance. pi_X must remain true until
    pi_Y is achieved. Example: "keep the river visible until you
    reach the tree" with pi_3 U pi_1.

Each constraint carries a polarity (negative or positive) stored in
a ConstraintInfo dataclass, which the monitor uses to label
constraints as "AVOID" or "MAINTAIN" in VLM prompts.

3a. LTL PROMPT DESIGN
-----------------------------
LTL_NL_SYSTEM_PROMPT includes documentation on constraint predicates:
  - Constraint predicates represent CONDITIONS TO ENFORCE, not goals.
  - Negative constraints: G(!pi_X) for unconditional avoidance,
    !pi_X U pi_Y for scoped avoidance.
  - Positive constraints: G(pi_X) for unconditional maintenance,
    pi_X U pi_Y for scoped maintenance.
  - Decision criteria: "never", "avoid", "stay away from" signal
    negative constraint predicates; "always keep", "maintain",
    "stay above" signal positive constraint predicates; "go to",
    "approach", "reach" signal goals.

LTL_NL_EXAMPLES_PROMPT includes worked examples:
  - Global avoidance: "Go to tree, then streetlight, never fly over
    building" with G(!pi_3)
  - Scoped avoidance: "Approach sculpture, stay away from red car
    until you reach it" with !pi_2 U pi_1
  - Multiple avoidance: "Go to park, then traffic light, never go
    near building A or B" with G(!pi_3) & G(!pi_4)
  - Global maintenance: "Fly to the landmark, always stay above
    10 meters" with G(pi_2)
  - Scoped maintenance: "Go to tree, then streetlight, keep river
    visible until tree" with (pi_3 U pi_1)
  - Mixed constraints: "Navigate to bridge, stay above treeline,
    never over highway" with G(pi_2) & G(!pi_3)

3b. AUTOMATON-BASED CONSTRAINT CLASSIFICATION
----------------------------------------------
After the Spot automaton is built, each predicate is classified as
goal or constraint, and each constraint's polarity is determined,
using a three-pass BDD query approach:

  Pass 1 (goal vs. constraint): A predicate is a CONSTRAINT if it
  never has a forward-progressing edge (dst != src) at any state.
  Goal predicates advance the automaton; constraint predicates do
  not.

  Pass 2 (polarity detection): For each candidate constraint, two
  BDDs are tested at the initial state:
    - "Only this predicate TRUE" BDD: if edges exist, the automaton
      accepts the predicate being true (positive candidate).
    - "All predicates FALSE" BDD: if no edges exist, the automaton
      rejects the predicate being false (positive confirmation).
  A constraint is classified as positive when the first BDD has edges
  and the second does not. Otherwise it defaults to negative.

  Pass 3 (re-check with positive constraints): Candidates classified
  as negative in pass 2 are re-checked using a BDD that sets the
  candidate TRUE alongside all identified positive constraints.
  If forward edges now appear, the candidate is actually a goal that
  was blocked by unsatisfied positive constraints in pass 1, and it
  is reclassified as a goal.

  Active constraint detection per state uses polarity-aware BDDs:
    - Negative: making the predicate TRUE produces no valid edge at
      the current state (the violation is rejected).
    - Positive: making the predicate FALSE (with other positive
      constraints set TRUE) produces no valid edge at the current
      state.
  When a scoped constraint expires (e.g., !pi_X U pi_Y after pi_Y,
  or pi_X U pi_Y after pi_Y), the violation BDD starts matching
  edges and the constraint is released.

  Goal checking (get_next_predicate, advance_state) also satisfies
  positive constraints in BDD queries, so the automaton correctly
  identifies forward edges even when positive constraints are present.

This approach is robust to any formula structure the automaton can
represent, with no regex parsing of the LTL formula string.

3c. GOAL ADHERENCE MONITOR CONSTRAINT HANDLING
-----------------------------------------------
The GoalAdherenceMonitor accepts a constraints parameter containing
ConstraintInfo objects (with description and polarity). When
constraints are present:

  - Constraints are injected into DIARY_GLOBAL_PROMPT and
    DIARY_CONVERGENCE_PROMPT after the subgoal line, labeled as
    "AVOID: ..." for negative or "MAINTAIN: ..." for positive.
  - The VLM JSON output includes a "constraint_violated": true/false
    field.
  - On violation: force_converge stops the drone, then the existing
    convergence correction flow issues a corrective command (e.g.,
    "move away from building B" for avoidance violations, or
    "ascend to restore altitude" for maintenance violations).
    Corrections share the same budget as normal convergence
    corrections.

The SubgoalConverter is NOT modified. Constraints are handled entirely
by the goal adherence monitor, since OpenVLA cannot process compound constraint
instructions.

3d. RUNNER INTEGRATION
----------------------------------------------
Before each subgoal, run_integration.py calls
planner.get_active_constraints() to extract ConstraintInfo objects
for constraints active at the current automaton state. These are
passed to the GoalAdherenceMonitor via _run_subgoal. Constraint status
(including polarity) is logged per subgoal in the result dict.


4. TASK DESIGN
======================================================================

4a. ENVIRONMENT
----------------
Simulation environment: DowntownWest (Unreal Engine, UnrealCV)
  - Outdoor suburban setting with diverse landmarks
  - Buildings (white building, red building, multi-story structures)
  - Vegetation (trees, hedges, parks)
  - Vehicles (black car, red car, white car, parked vehicles)
  - Infrastructure (streetlights, traffic lights, pergolas, sculptures)
  - People and dogs

NOTE: Currently only one simulation environment (DowntownWest) is
available. The "3 distinct environments" from the original plan
should be reframed as 3 distinct REGIONS within DowntownWest, each
with different landmark compositions and visual characteristics.
Select 3 non-overlapping regions of the map with sufficiently
different scenery (e.g., a residential area with trees and houses,
a commercial area with cars and buildings, a park area with
sculptures and pergolas).

4b. TASK CATEGORIES (5 tasks per region, 15 total)
---------------------------------------------------
Each task is a multi-step long-horizon instruction. Tasks are designed
to exercise different LTL operators and constraint types.

Category 1: PURE SEQUENTIAL (3 tasks)
  Simple ordered sequences (A then B then C).
  LTL: F pi_3 & (!pi_3 U pi_2) & (!pi_2 U pi_1)
  Purpose: Baseline difficulty. All conditions should handle these.
  Example: "Go past the first streetlamp, then turn right until you
  see the black car, then approach the black car."

Category 2: SEQUENTIAL WITH CONSTRAINTS (6 tasks)
  Ordered sequences where temporal constraints (avoidance or
  maintenance) must be enforced during execution.
  LTL: F pi_2 & (!pi_2 U pi_1) & (!pi_3 U pi_2) (avoidance)
       F pi_2 & (!pi_2 U pi_1) & G(pi_3) (maintenance)
  Purpose: THE KEY DIFFERENTIATOR. These tasks make the LTL formalism
  strictly necessary. Without formal constraint encoding, Condition 2
  (LLM planner) has no mechanism to enforce temporal constraints.
  Examples:
    - "Approach the tree from the right side, then go to the
      streetlight, but stay away from the red car until you reach
      the streetlight."
    - "Fly to the landmark, then the park, but always stay above
      10 meters altitude."

Category 3: BRANCHING / OR (3 tasks)
  Tasks with alternative paths (do A or B, then C).
  LTL: F pi_3 & (!pi_3 U (pi_1 | pi_2))
  Purpose: Tests automaton's ability to handle non-linear execution.
  Example: "Go to either the sculpture or the pergola, then approach
  the white building."

Category 4: COMPLEX MULTI-CONSTRAINT (3 tasks)
  Combine sequencing, mixed constraint types, and branching.
  LTL: F pi_3 & (!pi_3 U pi_1) & G(pi_4) & G(!pi_5)
  with concurrent avoidance and maintenance constraints.
  Purpose: Stress test the full system. Should maximally differentiate
  Condition 0 from all baselines.
  Example: "First approach the tree, then navigate to the traffic
  light, then land near the sculpture. Always stay above the
  treeline and never fly over the highway."

NOTE ON TASK BALANCE: At least 9 of the 15 tasks (60%) must include
meaningful temporal constraints (avoidance, maintenance, or both).
This is critical: without them, the gap between Condition 0 and
Condition 2 narrows, undermining the LTL contribution.

4c. TASK JSON FORMAT
---------------------
Each task JSON should include:
{
  "task_id": "region1_seq_neg_01",
  "category": "sequential_with_negative_constraints",
  "instruction": "Approach the tree from the right side, then go to
    the streetlight, but stay away from the red car until you reach
    the streetlight.",
  "initial_pos": [-7500, 848, 396, -177],
  "constraints_expected": [
    {"description": "stay away from the red car", "polarity": "negative"}
  ],
  "expected_subgoal_count": 2,
  "difficulty": "medium",
  "region": "region_1_residential",
  "notes": ""
}

4d. STARTING POSITION VARIATIONS
----------------------------------
Each task runs 3 times with different starting positions/orientations.
Variations:
  - Run 1: Default position (as specified in task JSON)
  - Run 2: Same position, rotated 90 degrees
  - Run 3: Offset 5-10 meters from default in a random direction
This tests robustness to initial conditions without requiring
entirely different scenarios.


5. EVALUATION METRICS
======================================================================

PRIMARY METRICS (reported in main results table):

M1. Task Success Rate
  - Definition: Fraction of episodes where the final long-horizon goal
    was fully completed (all subgoals achieved in correct order).
  - Measurement: Binary per episode. A task is successful only if all
    LTL subgoals were confirmed complete by the monitor (or by
    convergence in Condition 3).
  - Report: Mean +/- 95% Wilson confidence interval per condition.

M2. Constraint Adherence Rate
  - Definition: Of all episodes (successful or not), what fraction
    maintained all temporal constraints (both avoidance and
    maintenance) throughout execution?
  - Measurement: Binary per episode. Checked via goal adherence monitor
    constraint_violated flag (Conditions 0, 2, 4, 5, 6) or post-hoc
    trajectory analysis (Conditions 1, 3 which have no monitor).
  - Report: Mean +/- 95% Wilson CI per condition.
  - NOTE: For Conditions 1 and 3 (no monitor), constraint adherence
    must be evaluated post-hoc by passing the trajectory frames
    through a VLM and asking whether the constraint was violated at
    any point. This adds compute cost but is necessary for fair
    comparison.

M3. Subgoal Success Rate
  - Definition: Fraction of individual subgoals (across all episodes)
    that were successfully completed.
  - Measurement: Per-subgoal binary. Confirmed by monitor (Conditions
    0, 2, 4, 5, 6) or by convergence (Condition 3).
  - Report: Mean +/- 95% Wilson CI per condition.

DIAGNOSTIC METRICS (reported in analysis section):

M4. Supervisor Correction Rate (Rescue Rate)
  - Definition: Of all convergence events where the drone stopped
    prematurely (before subgoal completion), what fraction were
    successfully rescued by the supervisor's corrective commands?
  - Applicable to: Conditions 0, 2, 4, 5, 6 (not 1 or 3).
  - Measurement: (rescued convergences) / (total premature
    convergences)

M5. False Positive / False Negative Rate of Convergence Assessment
  - False Positive: Monitor declares subgoal complete but it is not
    (verified by post-hoc inspection or by failure of subsequent
    subgoal).
  - False Negative: Monitor issues correction when subgoal was
    actually complete (causes unnecessary additional steps).
  - Applicable to: Conditions 0, 4, 5, 6.
  - Measurement: Manual annotation of a subset of episodes (at least
    10 per condition) by reviewing frame sequences and monitor logs.

M6. Control Frequency / Latency
  - OpenVLA loop Hz: Frames per second of the control loop
    (excluding VLM checkpoint time in sync mode).
  - Monitor checkpoint latency: Wall-clock time per VLM checkpoint
    (local + global prompt pair).
  - Report: Mean and p95 per condition.
  - Purpose: Proves VLM monitoring does not block drone control in
    async mode, and quantifies overhead in sync mode.

M7. Token / Inference Overhead
  - Total VLM/LLM calls per episode (broken down by type: local,
    global, convergence, planning, subgoal conversion).
  - Total input tokens per episode.
  - Total output tokens per episode.
  - Estimated API cost per episode.
  - Report: Mean per condition.
  - Purpose: Proves computational feasibility. Addresses reviewer
    concern about running VLM monitoring continuously.

M8. Average Episode Length
  - Total steps (frames) per episode.
  - Total wall-clock time per episode.
  - Report: Mean +/- std per condition.
  - Purpose: Reveals whether monitoring adds overhead that slows
    task completion, even when it succeeds.


6. EXPERIMENTAL PROCEDURE
======================================================================

6a. PER-EPISODE PROCEDURE
---------------------------
1. Load task JSON (instruction, initial_pos, constraints).
2. Teleport drone to starting position. Wait for stabilization.
3. Reset OpenVLA model state.
4. Execute the pipeline for the active condition:
   - Condition 0: run_integration.py (full pipeline)
   - Condition 1: Single OpenVLA pass with raw instruction
   - Condition 2: LLM decomposition + goal adherence monitoring
   - Condition 3: run_ltl.py (LTL + blind execution)
   - Condition 4: Single-frame monitor variant
   - Condition 5: Grid-only monitor variant
   - Condition 6: Text-only global assessment variant
5. Log all frames, diary entries, monitor decisions, corrections,
   VLM call records, and trajectory poses.
6. Record final outcome: success/failure, stop_reason, subgoal
   completion status, constraint violations.

6b. EPISODE COUNT
------------------
15 tasks x 3 starting variations = 45 episodes per condition
7 conditions x 45 = 315 total episodes

6c. RANDOMIZATION
------------------
- Starting position variations are fixed across conditions (same
  3 positions per task for all 7 conditions) to enable paired
  comparison.
- Task presentation order is randomized per condition to avoid
  systematic ordering effects.

6d. MANUAL ANNOTATION
-----------------------
For M5 (false positive/negative analysis), manually annotate at least
10 episodes per condition (70 total) by reviewing:
  - Frame sequences around convergence events
  - Monitor decisions (complete vs. stopped_short)
  - Whether the decision was correct given the visual evidence
This can be done post-hoc from saved artifacts.

7a. TOKEN COST ESTIMATE (REVISED)
----------------------------------
These estimates need empirical validation before committing.

Per VLM checkpoint (local + global):
  - Local prompt: 1 system message + 2 images (224x224 each)
    Estimated: ~600 tokens (text) + ~1,700 tokens (2 images) = ~2,300
  - Global prompt: 1 system message + 3x3 grid image + diary text
    Estimated: ~800 tokens (text + diary) + ~4,000 tokens (grid image)
    = ~4,800
  - Total per checkpoint: ~7,100 input tokens

Per convergence assessment:
  - Similar to global: ~5,000 input tokens

Per episode (rough estimate):
  - ~15 checkpoints per subgoal x 3 subgoals avg = 45 checkpoints
  - 45 x 7,100 = ~320,000 input tokens
  - Plus ~5 convergence assessments x 5,000 = ~25,000
  - Plus planning + subgoal conversion: ~5,000
  - Total per episode: ~350,000 input tokens

IMPORTANT: These estimates must be validated empirically by running
2-3 test episodes and measuring actual token usage from the VLM call
logs. The image token count depends heavily on the VLM provider's
image encoding. Adjust the budget after validation.

Per-condition cost (at $2.50/MTok input for gpt-4o, or $10/MTok for
GPT-5.4):
  - gpt-4o monitor: 45 eps x 350K tokens x $2.50/MTok = ~$39
  - gpt-5.4 monitor: 45 eps x 350K tokens x $10/MTok = ~$158

Total for 7 conditions:
  - If all use gpt-5.4 monitor: ~$1,100
  - If Conditions 1, 3 use no monitor: savings of ~$316
  - Net estimate: ~$800-$1,100

NOTE: Condition 1 (naive) and Condition 3 (open-loop) do not use the

LLM (not VLM) for global checks, which is cheaper but still requires

8. STATISTICAL ANALYSIS PLAN
======================================================================

8a. PRIMARY COMPARISONS
-------------------------
For each primary metric (M1, M2, M3):
  - Report mean and 95% Wilson confidence interval per condition.
  - Pairwise comparison: Condition 0 vs. each baseline using Fisher's
    exact test (for binary outcomes) or McNemar's test (for paired
    episodes with matched starting positions).
  - Significance threshold: p < 0.05 with Bonferroni correction for
    6 pairwise comparisons (adjusted threshold: p < 0.0083).

8b. EFFECT SIZE
----------------
With n=45 per condition, a Fisher's exact test can detect a difference
of ~25 percentage points (e.g., 85% vs. 60%) at 80% power with
alpha = 0.05. Design tasks to produce large effect sizes. If pilot
runs suggest smaller gaps, increase episode count or reduce the
number of conditions.

9. KEY ASSUMPTIONS AND LIMITATIONS
======================================================================

A1. Low-level UAV safety is handled separately.
    Physical collision avoidance, altitude bounds, and geofencing are
    assumed to be implemented via control barrier functions (CBF) or
    depth-map heuristics. This is standard in UAV systems and is
    orthogonal to the task-level planning contribution of this work.
    State this as a scope limitation in the paper.

A2. Single simulation environment.
    All experiments run in DowntownWest. Frame this as evaluating
    across 3 diverse regions of a realistic suburban environment.
    Acknowledge single-environment limitation in the paper and
    suggest real-world validation as future work.

A3. OpenVLA as the fixed low-level policy.
    The system is designed to be policy-agnostic. OpenVLA is one
    instantiation. The contribution is the LTL + monitoring framework,
    not the specific VLA model.

A4. VLM accuracy for constraint monitoring.
    The goal adherence monitor relies on VLM visual understanding to detect
    constraint violations. VLM errors (failing to notice the drone
    is near a forbidden landmark) will propagate as undetected
    violations. Quantify this in M5.

A5. Simulation fidelity.
    UnrealCV provides photorealistic rendering but lacks dynamic
    obstacles, weather variation, and wind disturbances present in
    real-world flight.

11. PAPER FRAMING NOTES
======================================================================

- Frame Condition 2 as a SOTA-representative baseline. Cite SayCan,
  Inner Monologue, Code-as-Policies as related LLM-planner methods
  and argue Condition 2 captures their core approach (LLM text
  decomposition + sequential execution) applied to UAV navigation.
  Note that none of these prior methods target UAV control with LTL.

- The strongest story comes from the gap between Condition 2 and
  Condition 0 on constraint tasks (both avoidance and maintenance).
  This is the unique contribution of LTL formalism over ad-hoc LLM
  planning.

- The second strongest story comes from Condition 3 vs. Condition 0.
  This proves that even with perfect planning (same LTL), execution
  monitoring is critical.

- Conditions 4, 5, 6 are design ablations that justify architectural
  choices. They can go in an ablation subsection or appendix if
  space is tight.

- Report M7 (token overhead) prominently to preempt feasibility
  concerns. Show that monitoring adds <X% wall-clock overhead in
  async mode.
